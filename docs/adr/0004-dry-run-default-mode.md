# ADR 0004: Dry-Run (Paper Trading) as Default Mode

**Status:** Accepted
**Date:** 2026-03-18
**Deciders:** Engineering Lead

---

## Context

Sentinel Apex is an automated system that submits real orders to financial venues. A misconfigured risk limit, a strategy engine bug, a data feed error, or an untested code path can result in material capital loss within seconds. The system must have a reliable mechanism to develop, test, and validate behavior without placing capital at risk.

The question is not whether to support a simulation mode — that is a given. The question is how to structure the simulation mode relative to live trading:

**Option A: Live by default, simulation as an explicit opt-out.** The system trades live unless a flag is set to disable live trading. New deployments default to live.

**Option B: Simulation by default, live as an explicit opt-in.** The system runs in paper-trading mode unless a specific configuration is provided to enable live trading. New deployments default to paper.

**Option C: Strict parity simulation with separate data schemas.** Simulation mode is not a "lite" version of live. The full execution pipeline — signal generation, risk checks, order submission, fill processing, reconciliation — runs identically. The only difference is whether the venue adapter submits to the real venue or to a simulated fill engine. Both modes persist state to the database with strict separation between schemas.

The team additionally considered whether paper-trading records should be stored in the same database as live records (with a `is_paper: boolean` flag) or in entirely separate schemas.

---

## Decision

**Dry-run (paper trading) is the default mode of operation.** Live execution requires explicit, layered opt-in. The system implements Option C with separate database schemas for paper and live state.

### Default Behavior

- On first startup with no explicit trading mode configuration, the system starts in `paper` mode
- A prominent log line is emitted at startup: `[TRADING MODE: PAPER] Live execution is disabled. Set TRADING_MODE=live to enable.`
- All signals are generated, all risk checks run, all orders are submitted — but to the simulated fill engine, not to real venues
- All positions, fills, and audit events are recorded to the `paper` database schema

### Live Mode Activation

Live trading requires all of the following to be simultaneously true:

1. **Environment variable:** `TRADING_MODE=live` is set in the process environment
2. **Confirmation token:** `LIVE_TRADING_CONFIRMATION=<token>` is set, where `<token>` is a dated HMAC token generated by the system's confirmation endpoint within the past 15 minutes
3. **Production flag:** `NODE_ENV=production` is set (prevents accidental live trading in development environments)

If any of these conditions is not met, the system starts in paper mode regardless of other settings. This is intentional: it is better to paper trade accidentally than to live trade accidentally.

The confirmation token flow:
```
1. Operator calls POST /admin/live-trading/confirm-token (authenticated)
2. System generates a time-limited HMAC token and returns it
3. Operator sets LIVE_TRADING_CONFIRMATION=<token> in environment
4. System validates token on startup; if valid, enables live mode
5. Token is single-use; a restart requires a new token
```

### Code-Level Separation

The paper/live separation is implemented at the lowest level of the execution stack — the venue adapter factory:

```typescript
// packages/execution/src/adapter-factory.ts

export function createVenueAdapters(config: Config): VenueAdapterMap {
  if (config.tradingMode === 'live') {
    return {
      drift: new DriftAdapter(config.drift),
      binance: new BinanceAdapter(config.binance),
      // ...
    };
  }
  return {
    drift: new SimulatedVenueAdapter('drift', config.simulation),
    binance: new SimulatedVenueAdapter('binance', config.simulation),
    // ...
  };
}
```

No other code in the system contains a conditional on `tradingMode`. Strategy engines, risk checks, the allocator, and the execution coordinator are unaware of whether they are running in paper or live mode. This is the key design requirement: paper mode must exercise the exact same code paths as live mode.

### Simulated Fill Engine

The `SimulatedVenueAdapter` implements the full `VenueAdapter` interface:

- `submitOrder()`: Records the order, computes a simulated fill using mid-price ± slippage model, returns a synthetic fill
- `getPositions()`: Returns the in-memory simulated position state
- `getBalances()`: Returns simulated balances computed from fill history
- `getFundingRate()`: Passes through to the real venue adapter (funding rate data is always real, not simulated)
- `getOrderBook()`: Passes through to the real venue adapter (market data is always real)

Slippage model: `fillPrice = midPrice × (1 + slippageFraction × side)` where `slippageFraction` is configurable per venue and defaults to the venue's historical median taker slippage.

Funding payments in paper mode are computed using real historical funding rates applied to simulated position sizes at each funding interval.

### Database Schema Separation

Paper trading state and live trading state are stored in separate PostgreSQL schemas:

- `paper.*` — all tables for paper trading (positions, orders, fills, etc.)
- `live.*` — all tables for live trading

The application connects to a schema-qualified database session. A live-mode bug cannot corrupt paper-mode records, and paper-mode test data cannot contaminate live NAV calculations.

The `audit_events` table exists in both schemas. Paper audit events and live audit events are fully independent.

### Promotion from Paper to Live

There is no automated migration of paper positions to live. When live trading is first enabled, the system starts with a clean live state: zero positions, zero fills, NAV equal to deposited capital. Any open paper positions are left in the paper schema as historical records.

---

## Consequences

**Positive:**
- New code changes can be validated in paper mode against live market data before any capital is deployed
- The identical code path in paper and live mode means paper test results are highly predictive of live behavior (the only difference is fill price realism, which the slippage model approximates)
- The separate database schemas prevent any possibility of paper state contaminating live portfolio accounting
- The layered live mode activation (env var + token + NODE_ENV) makes accidental live trading in development or CI environments essentially impossible
- Operators build familiarity with system behavior and the ops dashboard before capital is at risk

**Negative:**
- The simulated fill engine cannot perfectly model all venue behaviors: order rejection edge cases, partial fill sequencing, and fee tier differences may behave differently in live mode
- Paper trading results are not guaranteed to match live results; the system explicitly does not claim that paper NAV is an accurate predictor of live NAV due to simulation assumptions
- The dual-schema approach doubles storage for certain tables; acceptable at expected data volumes

**Neutral:**
- CI and automated tests always run in paper mode; there is no test mode distinct from paper mode
- The ops dashboard displays a prominent banner indicating whether the system is in paper or live mode
- Switching from live back to paper requires closing all live positions first; the system will not enter paper mode while live positions exist
